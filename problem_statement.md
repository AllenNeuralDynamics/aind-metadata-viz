End-to-End Session Observability: A Phased Proposal 

March 11 draft written by Claude Opus 4.6 in response to a discussion thread w/ Doug Ollerenshaw. Includes minor posthoc edits by Doug. 

 

1. The Problem 

When a scientist runs a data collection session at AIND, there is no single place to answer the question: where is my data right now? 

A session passes through multiple systems on its way from the rig to the cloud: acquisition software, watchdog, the Data Transfer Service (DTS), on-premises VAST storage, S3, DocDB metadata registration, and Code Ocean processing pipelines. Each system tracks its own piece, but no system tracks the whole journey. When something fails, it can take hours of manual detective work to figure out where and why. 

This problem has been raised multiple times, but most recently in a project request from Cindy for a “Lifetime Session Tracker” (aind-scientific-computing #463). Multiple efforts have been started — logging standards, a Grafana proof-of-concept, an Acquisition Database proposal — but none have delivered a working solution to users. 

2. Why Progress Has Stalled 

After reviewing the relevant issues, proposals, and discussion threads, it seems that the major problem is that at least four distinct sub-problems are being conflated, and different people are optimizing for different ones. 

Sub-problem A: No persistent session identifier 

Currently, the asset name (e.g. behavior_746346_2026-03-11_09-00-00) is generated by the DTS during upload. Before that point, there is no stable key to correlate logs across systems. This is the “token server” concept that has been discussed but not built. 

Sub-problem B: Pre-watchdog failures are invisible 

Most session failures occur before the watchdog picks up the data, as Bruno has repeatedly emphasized (see aind-scientific-computing #568). These failures are completely invisible to any downstream tracking system. Solving this requires rig-side instrumentation, which is Bruno and SIPE’s domain. 

Sub-problem C: No unified entry point across existing tools 

The DTS already has a job status page with detailed task-level tracking and log access (more on this below). The metadata portal already has an asset viewer showing raw and derived assets. But these are separate tools. A scientist has to know to look in both places and manually cross-reference between them to understand whether a session made it all the way through. 

Sub-problem D: Operational metadata and scheduling 

Cindy’s team needs WaterLog integration, session scheduling, and experiment-level metadata tracking (IACUC protocols, etc.). These are real needs, but they are fundamentally different from pipeline observability. They have been mixed into the session tracking discussion because the Acquisition Database proposal (aind-scientific-computing #594) began absorbing scheduling features. 

The result 

The Acquisition Database proposal tries to be a token server, a metadata validator, a scheduling database, and a DataDescription generator all at once. Clark commented that what was described was not what they understood the system to be. Reviewers (Bruno, Saskia, Dan) all flagged scope concerns in comments on the proposal document. The proposal is stuck in design review. 

Meanwhile, the logging standard work (aind-log-utils PR #25, merged March 10, 2026) defines a JSON format and field names, but no services have adopted it yet, and the acquisition_name field in the spec has no value to populate until the token server exists. 

Ahad’s Grafana dashboard proof-of-concept demonstrated that log data can be aggregated, but the feedback in aind-scientific-computing #568 made clear that what users need is a status table (“which stage is my session in?”), not log visualizations. The dashboard also had authentication issues that made it hard for stakeholders to access. 

3. What Already Exists (and What’s Actually Missing) 

Before proposing new work, it’s worth being precise about what we already have, because the existing tools are more capable than is sometimes appreciated. 

The DTS Job Status page 

Available at http://aind-data-transfer-service/jobs, this provides: 

A filterable table of all DTS jobs with filters for status, date range, asset name, job type, and DAG ID. 

A task-level drill-down for each job showing every step in the pipeline (e.g. check_s3_folder_exists, gather_preliminary_metadata, compress_data, upload_data_to_s3, register_data_asset, run_codeocean_pipeline, etc.) with individual success/fail/skipped status for each. 

Full Code Ocean pipeline tracking. The DTS doesn’t just trigger CO pipelines — it monitors them through to completion. For multi-modality jobs like vr_foraging_fiber, both the behavior and FIP pipelines appear as separate tasks (run_codeocean_pipeline with map_index 0 and 1), each with their own status. Which pipelines run for a given job type is defined in the Job Parameters configuration. 

Log access for every task, including full Airflow logs with error tracebacks for failures and Code Ocean pipeline execution details. 

Deep-linkable task views. The URL http://aind-data-transfer-service/job_tasks_table?dag_id=transform_and_upload_v2&dag_run_id=<job_id> renders the task list for any job directly. This means external tools can link into the DTS drill-down without rebuilding it. See, for example, http://aind-data-transfer-service/job_tasks_table?dag_id=transform_and_upload_v2&dag_run_id=manual__2026-02-27T01:19:43.324824%2B00:00 

For a workflow like VR Foraging + FIP, the DTS creates one job per session that handles everything: uploading the raw data (which contains all modalities together), compressing videos, registering the raw asset, and triggering both the behavior and FIP processing pipelines. The raw asset ends up on S3 with all modalities bundled together, and the two Code Ocean pipelines each produce their own derived asset. 

The Asset Viewer 

Available at https://data.allenneuraldynamics.org/assets, this shows: 

All registered assets (raw and derived) with subject ID, acquisition time, project, modalities, and data level 

Links to Code Ocean, metadata records, and QC 

Filterable by subject ID, project, and other fields 

For example, a single VR Foraging + FIP session for subject 822683 appears in the asset viewer as one raw asset (modalities: behavior, behavior-videos) and two derived assets (one for behavior, one for fib). 

So what’s actually missing? 

Given how much the DTS already tracks, the gap is narrower than it might first appear. What’s genuinely missing is: 

1. A project-filtered, session-level entry point. The DTS shows 1,349+ jobs across all projects in the last two weeks. A scientist on the Patch Foraging team has to filter by asset name or job type to find their sessions. There is no view that starts from “show me Cognitive Flexibility in Patch Foraging sessions this week” and surfaces just those sessions with their end-to-end status. 

2. Confirmation that derived assets landed in DocDB. The DTS tracks whether the Code Ocean pipelines ran successfully as tasks, but a scientist also wants confirmation that the derived assets are actually registered and accessible in DocDB. Connecting DTS job completion to DocDB asset records closes this loop. 

3. Visibility for sessions that never made it to the DTS. If a session failed during acquisition or was never picked up by the watchdog, it simply doesn’t exist in the DTS. There’s no record of the absence. This is the fundamental gap that the token server (Phase 1) and rig-side observability (Phase 2) would address. 

A note on the DTS 14-day limit: The DTS API currently enforces a hardcoded two-week query window (a validator in the Python code rejects requests for jobs older than 14 days). This is a limitation of the DTS viewer, not of the underlying Airflow database, which retains data much longer. For Phase 0, this is manageable: DocDB (which has no time limit) serves as the primary data source for sessions, and the DTS API provides job status and deep-link URLs for recent sessions. For sessions older than 14 days, the viewer would still show DocDB asset records but would not be able to link to the DTS task drill-down. If this becomes a pain point, relaxing the DTS time limit is a small code change in aind-data-transfer-service. 

4. Proposed Approach: Three Phases 

Rather than building the complete system before delivering anything, this proposal defines three phases. Each phase delivers standalone value and does not depend on subsequent phases to be useful. 

Phase 0: Session Status View (no changes to existing systems) 

Goal: Build a lightweight page that gives scientists a project-filtered, session-level view of their data, joining DTS job status with DocDB asset records and linking into existing tools for drill-down. 

Key principle: Don’t rebuild what exists. The DTS task drill-down and log viewer already work well. The asset viewer already works well. What’s needed is a thin integration layer that connects them and surfaces the gaps. 

What it looks like 

A page on the AIND metadata portal (the same platform that hosts the asset viewer and tools like the fiber viewer). The page would show: 

A table where each row is a session, filtered by project and date range 

Columns showing status at key stages: DTS job status (overall), raw asset registered in DocDB, derived asset(s) registered in DocDB 

Status indicators: ✅ completed, ❌ failed, ⏳ in progress, blank for not yet reached 

Direct links to existing tools: Clicking the DTS status cell opens the deep-linkable DTS task view for that job (via http://aind-data-transfer-service/job_tasks_table?dag_id=...&dag_run_id=...), where the user can see every task, every CO pipeline result, and every log. Clicking an asset cell opens the metadata record or Code Ocean link. 

How it stays up to date: The page does not maintain its own database or require any event hooks. It queries the DTS API and DocDB live on every page load, just as the existing asset viewer and fiber viewer do. A scientist bookmarks a URL like ?project=Cognitive+flexibility+in+patch+foraging&days=7, and every time they load that page, they see the current state of their sessions. New sessions appear as soon as they're registered in DocDB or picked up by the DTS. No synchronization infrastructure is needed. 

The core value: a scientist opens this page, selects their project and date range, and immediately sees which sessions completed successfully end-to-end, which ones had failures (with a direct link to see where and why), and which ones are still in progress. No hunting through 1,349 jobs or cross-referencing separate tools.

Phase 1: Session Token Server 

Goal: Extend tracking backward to session start by creating a lightweight token registration service. This enables pre-DTS tracking and provides a stable correlation key for rig-side logs. 

What to build 

A single REST API endpoint (could live in aind-metadata-service) that accepts a subject_id and username, generates a UUID, stores it in a Dataverse table with a timestamp, and returns the UUID. That is the entire scope. No validation against other tables. No DataDescription generation. No scheduling. 

Per Dan Birman’s suggestion: the UUID is not the asset name. The human-readable asset name continues to be generated at upload time by the DTS (and should eventually use build_data_name from aind-data-schema-models, per aind-data-transfer-service #335). The UUID is purely a correlation key. 

Integration points 

Each integration is a small change in the relevant repository: 

Rig acquisition software: Call the token API at session start. Include the UUID in local logs. (Owner: Bruno / SIPE) 

Watchdog: Pass the UUID through to the DTS when triggering upload. (Owner: SIPE) 

DTS: Accept an optional session_token field. Include it in Airflow DAG run metadata. (Owner: Jon’s team) 

Phase 0 session view: Update to query by session token in addition to asset name. (Owner: Arielle’s team) 

Why this is harder than it sounds 

Even though the token server itself is simple, it touches rig acquisition code (Bruno/SIPE’s domain), the watchdog (SIPE), the DTS (Jon’s team), and the metadata service (Arielle’s team). Getting all those teams to coordinate changes, review PRs, and deploy in sequence is the kind of cross-team work that can stall for months. This is why Phase 0 is designed to deliver value without requiring any of this coordination. 

Estimated effort 

API itself: 1 engineer, a few days 

Integration across systems: Requires coordination across 3 teams; calendar time likely 4–6 weeks 

Shape 

Phase 2: Full Rig-Side Observability 

Goal: Capture pre-watchdog events (acquisition start, acquisition failure, data save confirmation) so that sessions that fail before reaching the DTS become visible. 

This phase requires rig software to emit structured log events using Jessy’s logging standard (aind-log-utils), with the session token as a required field. These logs get sent to Loki. The session status view adds columns for rig-side stages and queries Loki for their status. 

This is where the Grafana/Loki infrastructure becomes essential — but only for rig-side logs. DTS and downstream stages continue to be tracked through their existing APIs. 

Estimated effort: Depends on the number of rig software platforms that need instrumentation. Start with one (Patch Foraging rigs) and expand.

6. Related but Separate Work 

Several important needs have come up alongside the session tracking discussion. These are real priorities for the Behavior Platform, but they are distinct efforts that should be planned and tracked independently rather than bundled into session observability: 

Session scheduling: Determining which mice run on which rigs and when. This is an operational workflow need that Cindy’s team has described. It could be built on Dataverse and is related to the scheduling concepts in aind-scientific-computing #594, but it does not require — and should not wait for — the session tracking system described here. 

Operational metadata (IACUC protocols, experiment descriptors): Tracking metadata like ethics IDs, funding sources, and experiment types that are needed at upload time but aren’t currently captured at the rig. The Behavior Platform Dataverse needs document (linked by Cindy) describes these needs well. This work interfaces with Dataverse table design and the metadata service, and can proceed in parallel. 

WaterLog integration: Connecting water restriction tracking to the data platform. Given Clark’s recent progress, this seems close to done, but it is not a blocker for progress on the core session tracking concept. 

These are all worth pursuing. Separating them from session observability lets each one be scoped, staffed, and delivered on its own timeline without blocking the others. 

7. Key Decisions for Reviewers 

Phase 0 approach: Does building a session-level view on the metadata portal (joining DTS status with DocDB records, linking into existing drill-down tools) seem like the right first step? 

Phase 1 scope for aind-scientific-computing #594: Can the Acquisition Database proposal be formally descoped to just the token server, with scheduling and validation deferred to separate issues? 

Token format: UUID (as Dan Birman suggested) vs. the current asset name format? This proposal recommends UUID, with the human-readable name generated at upload time. 

Phase 0 ownership: Who builds it? This is a small, self-contained project (1–2 weeks for one engineer) with no cross-team dependencies. Can we (scientific computing) attempt to build a quick prototype for feedback, rather than continue to circle around conceptual discussions that inevitably lead to scope creep and indecision?
